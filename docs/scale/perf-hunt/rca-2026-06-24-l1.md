# 🚧 RCA — perf-hunt false positive: parallel L1 aggregation (2026-06-24)

**What happened:** The perf-hunt workflow reported "parallelize L1 aggregation" as a confident
**KEEP: −2.95 ms wall p99 / −1.52 ms l1_aggregation p99 @ 1M** (5 seeds, phase win 15× noise). It was
implemented (TDD, byte-identical), reviewed, committed (`a0aa4e8`), and merged to `main`. On the
author's real hardware it was a **REGRESSION**: serial L1 rebuild ~3 ms (1M holds 2–3 ms under the
50 ms budget); parallel ~6 ms, **tripping the budget**. Reverted in `bd38a2b`.

The byte-identical correctness was never wrong. The **performance claim** was wrong. This RCA is about
why the harness reported a win that did not exist.

## Empirical confirmation (the deciding experiment, run 2026-06-24)

A clean back-to-back A/B in the **headless** lab — serial then parallel, adjacent runs, clean tree,
1M / 5 seeds — settles which cause dominated:

| | wall p99 | l1_aggregation p99 |
|---|---|---|
| serial   | 46.8 ms ✓ | 4.4 ms |
| parallel | 57.2 ms ✗ (busts budget) | 14.0 ms |
| verdict  | **DITCH** (+10.4 ms wall) | +9.6 ms |

**The headless lab measures the regression correctly and sensitively.** So cause #1 below is
**REFUTED** — the lab is sound; it does NOT need a frontend or production load to catch this. The
original hunt reported **−2.95 ms**; reality is **+10.4 ms** — a 13 ms error in the wrong direction
that drift cannot produce. That can only mean the hunt **never measured the parallel code in
isolation** (dirty-tree contamination). **Cause #3 is the primary cause.** The clean back-to-back A/B
*is* the fix, and it returned the correct verdict on the first try.

## Root causes

### 1. REFUTED — "the headless lab is unrepresentative"
Plausible on paper (the lab runs headless on idle cores; production shares cores with render/IPC/
Electron), but the empirical A/B above shows the clean headless lab catches the +10 ms regression
outright. The lab's isolation was never the problem here. *Keep this in mind for future
contention-sensitive changes, but it is NOT what happened this time.*

### 2. CONTRIBUTING — the algorithm had no margin to begin with
Bit-identity (no float-add reorder) forced **output-partitioning**: each Rayon task owns disjoint L1
rows, so every one of ~167 L1 row-bands **re-scans the ENTIRE L0 non-empty list** — `O(L1_height ×
non_empty)`, roughly **100× the scan work** of the serial single pass. That only nets positive when
cores are abundant and idle (the lab). Any change whose payoff depends on free-core availability is
inherently fragile, and the lab is exactly the environment that hides the fragility.

### 3. PRIMARY (confirmed) — measurement methodology
- **Pinned single baseline, no back-to-back A/B.** The baseline was measured once at the start of a
  ~35-min run; every candidate was diffed against it. Machine drift over that window is uncontrolled —
  and indeed **4 of 5 candidates showed spurious ~−1 ms wall deltas**, the signature of drift, not real
  wins.
- **Imperfect patch isolation.** The ledger note on another candidate — *"Candidate diff was already
  present in working tree at task start"* — is direct evidence that the `git apply -R` revert between
  serial candidates did not always leave a clean tree. A measurement taken on a contaminated tree is
  not evidence.

### 4. PROCESS — merged on an unreproduced perf number
The change was committed on the strength of the hunt's number. The **byte-identical test guards
correctness, not speed**, and the author explicitly flagged — then merged past — *"I haven't re-measured
my exact code."* A perf claim must be **independently reproduced** before merge. The human A/B is what
caught this; it should have been a required gate, not a lucky afterthought.

## Fixes (workflow hardening)

1. **Parallelism/threading changes are lab-unreliable — flag and confirm in the real app.** Any
   candidate that adds Rayon/threads gets marked "lab cannot judge contention; confirm under
   production load" and is never auto-recommended on lab numbers alone.
2. **Back-to-back A/B per candidate.** Measure baseline and candidate adjacently in the same serial
   slot; report the delta from that local pair, not from a stale pinned baseline. Kills drift.
3. **Hard patch isolation.** Assert `git status --porcelain` is empty before each candidate build;
   abort/flag the candidate if not. Never measure on a dirty tree.
4. **Replicate the final KEEP.** Re-measure any KEEP a second time, independently, before recommending
   merge. A win that doesn't reproduce twice isn't a win.
5. **Human perf-confirmation gate before merge** for anything touching the hot loop — exactly the A/B
   that exposed this one.

## Status of the fixes
- #2 (back-to-back A/B) + #3 (clean-tree guard): **implemented** in `perf-hunt.mjs` and **proven** —
  the clean back-to-back A/B above returned the correct DITCH verdict on the first try.
- #4 (replicate the KEEP before recommending): **implemented**.
- #1/#5 (contention caveat for parallelism changes; human perf-confirm before merge): retained as
  process guidance; the headless lab is trusted for the algorithmic/cache/alloc majority.
